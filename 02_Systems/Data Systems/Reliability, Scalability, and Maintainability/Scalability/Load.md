# Load

Date: 2026-06-05

---

## 1. Definition

Load describes the demand placed on a system.

It represents the amount of work a system must perform and is typically measured using one or more load parameters.

Common load parameters include:

- Requests per second (RPS)
    
- Concurrent users
    
- Read-to-write ratio
    
- Data volume
    
- Cache hit rate
    
- Network traffic
    
- Transactions per second (TPS)
    

Load is not a single number; different systems are stressed in different ways.

---

## 2. Why It Matters

Scalability can only be discussed in relation to a particular load.

Before improving a system, engineers must understand:

- What kind of work the system performs
    
- Which load parameters are increasing
    
- Which resources become bottlenecks under load
    

Understanding load allows engineers to design systems that continue performing well as demand grows.

---

## 3. Key Ideas

### Load Parameters

Load is measured using parameters that describe the workload.

Examples:

|System|Important Load Parameter|
|---|---|
|Web Server|Requests per second|
|Database|Queries per second|
|Social Network|Active users|
|Cache|Hit rate|
|Streaming Platform|Data transferred per second|

Different systems may experience very different loads even if they have the same number of users.

### Bottlenecks

As load increases, some resource eventually becomes the limiting factor.

Common bottlenecks include:

- CPU
    
- Memory
    
- Disk I/O
    
- Network bandwidth
    
- Database throughput
    

Identifying bottlenecks is critical for scaling a system effectively.

### Load Patterns

Systems rarely experience constant load.

Common patterns include:

- Daily traffic cycles
    
- Sudden traffic spikes
    
- Seasonal demand
    
- Viral growth events
    

Systems must often be designed for peak load rather than average load.

## 4. Examples

### Twitter Timeline Problem

![[pic_twitter_SQL.png]]

In the original approach, tweets were stored centrally and timelines were generated by querying all followed users when a page was loaded.

This worked well when load was relatively small.

---

![[pic_twitter_method_2.png]]

As Twitter grew, timeline generation became more expensive.

The second approach precomputed timelines by pushing tweets into followers' caches whenever a tweet was posted.

This shifted work from reads to writes.

---

### Hybrid Approach

Neither approach is optimal in all situations.

For users with millions of followers, pushing updates to every follower becomes expensive.

For users with few followers, precomputing timelines is efficient.

Modern systems often combine both approaches to balance load across the system.

---

## 5. How Load Is Managed

Common techniques for handling increasing load include:

### Vertical Scaling

Increase the resources of a single machine.

Examples:

- Faster CPU
    
- More memory
    
- Larger storage
    

### Horizontal Scaling

Distribute work across multiple machines.

Examples:

- Load balancing
    
- Replication
    
- Partitioning
    

### Elastic Scaling

Automatically add or remove resources based on demand.

Common in cloud environments.

### Architectural Optimization

Redesign components to distribute work more efficiently and eliminate bottlenecks.

---

## 6. Tradeoffs

Different architectures handle load differently.

For example:

- Precomputing timelines improves read performance but increases write cost.
    
- Generating timelines on demand reduces write cost but increases read cost.
    

There is no universal architecture that performs best under every type of load.

System design involves selecting tradeoffs that match expected usage patterns.
